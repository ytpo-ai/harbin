# 后端开发建议规范

> 本规范定义了项目后端开发中的结构与维护建议，供开发人员在进行后端开发时参考遵循。

## 1. 后端大文件维护规则

### 1.1 Agents 模块职责拆分（强制）

- 当需求涉及 Agents 时，`CRUD` 逻辑与`执行`逻辑必须拆分，禁止继续聚合在同一大文件中。
- `CRUD` 逻辑（创建、查询、更新、删除）应归属资源管理 Service/Controller。
- `执行` 逻辑（运行、调度、编排、执行态处理）应归属执行链路 Service/Executor。
- 禁止在单个 Service 中同时承载完整 `CRUD + 执行` 主流程，避免职责耦合和文件持续膨胀。

### 1.2 维护目标

- 控制文件体积，避免出现难以维护的后端大文件。
- 降低模块耦合，提升回归测试与故障定位效率。
- 保证后续功能扩展时可按职责独立演进。

### 1.3 落地要求

- 新增功能优先在对应职责文件扩展，不得回填到历史聚合大文件。
- 评审时若发现 `CRUD` 与`执行`逻辑混写，应作为结构问题要求拆分后合并。
- 相关改造应同步补齐或更新对应测试，确保行为不回归。

## 2. TypeScript Import 顺序规则

### 2.1 分组顺序（强制）

- `1) Node 内置模块`：如 `crypto`、`path`。
- `2) 第三方包`：如 `@nestjs/*`、`mongoose`、`uuid`。
- `3) 跨应用/共享层`：`@libs/*` 优先，其次 `@legacy/*`。
- `4) 当前领域模块（alias）`：如 `@agent/modules/*`、`@agent/schemas/*`。
- `5) 当前目录相对路径`：`./*`、`../*`。

### 2.2 书写规范

- 同组内按字母序排序。
- 组与组之间保留一个空行。
- `import type` 与 value import 分开（可选 `type` 统一置前或置后，但需团队固定并保持一致）。
- 全项目统一使用半角单引号 `'`，禁止使用弯引号（如 `’`）。

### 2.3 执行建议

- 推荐通过 ESLint（如 `import/order` 或 `simple-import-sort`）固化规则，并启用自动修复。

---

> 本规范用于沉淀后端结构约束与 import 组织规则，后续同类后端模块默认参照执行。
