# OpenCode 多环境协同（Local + ECDS）技术设计

## 1. 设计背景

研发执行存在多个环境：

- 本地开发环境（Local）
- ECDS 开发环境（Server）

目标是在不共享同一数据库直写的前提下，实现跨环境协同执行、统一观测与统一分析。

## 2. 设计目标

1. 打通 Local 与 ECDS 的执行数据隔离问题。
2. 保持中心数据一致性与可审计性。
3. 保持边缘环境离线可执行与在线可补偿。

## 3. 架构方案

采用 **中心控制面 + 边缘执行面 + 同步汇聚层**：

1. 中心控制面（Control Plane）
   - 运行在中心服务（建议 ECDS）
   - 管理计划、任务、审批、配额、EI 分析
2. 边缘执行面（Execution Plane）
   - Local/ECDS 节点执行 OpenCode 任务
   - 保存本地临时执行态与待同步队列
3. 同步汇聚层（Ingest）
   - 统一接收边缘上报事件
   - 幂等去重 + 顺序校验 + 脱敏校验 + 审计落库

## 4. 标识与主键设计

所有同步数据必须包含：

- `envId`：环境标识（`local` / `ecds`）
- `nodeId`：执行节点标识（如 `local-macbook-01`）
- `runId`：跨系统运行主键
- `eventId`：事件幂等键
- `sequence`：事件顺序键

约束：

- 幂等主键：`(runId, eventId)`
- 顺序索引：`(runId, sequence)`
- 同 run 允许跨环境观测，但仅一个“主执行源”可写终态

## 5. 同步协议

## 5.1 上报接口

- `POST /engineering-intelligence/opencode/ingest/events`

请求包含：

- `syncBatchId`
- `envId/nodeId`
- `run` 基本信息
- `events[]` 事件明细（A 方案）
- `stats/costRaw`

## 5.2 写入规则

1. 先验签（节点签名 + Gateway 内部上下文）。
2. 幂等去重（同 `runId+eventId` 重复写入返回幂等成功）。
3. 顺序校验（`sequence` 连续升序，不连续拒收并返回可重试）。
4. 脱敏校验（payload 敏感字段必须 masked）。

## 5.3 补偿机制

1. 边缘离线时写本地待同步队列。
2. 恢复后按 `sequence` 顺序回放。
3. 中心拒收后按错误码策略重试或人工介入。

## 6. 冲突与一致性策略

### 6.1 终态冲突

- 同一 `runId` 如出现多个终态写入：
  - 以首个有效终态为准
  - 后续终态标记 `conflicted` 并进入审计队列

### 6.2 多环境并发执行

- 允许不同 run 在不同环境并发。
- 不允许同 run 被两个环境同时领取为主执行者。
- 主执行租约（lease）建议字段：`leaseOwnerNodeId/leaseExpireAt`。

## 7. 安全模型

1. 边缘节点使用独立凭证（按 `nodeId` 颁发）。
2. 支持节点吊销、轮换与最小权限。
3. 禁止边缘节点直连中心核心分析集合。

## 8. 观测与运维

核心指标：

- 节点上报成功率（按 `envId/nodeId`）
- 批次重试次数与失败率
- 序列缺口率（sequence gap）
- 冲突终态数量

排障路径：

1. 查 run 同步状态（pending/synced/failed）
2. 查节点健康（最后心跳、失败批次）
3. 查审计（冲突、拒收原因、重试轨迹）

## 9. 为什么不采用“所有环境同库直连”

1. 本地环境风险会直接污染中心数据。
2. 权限边界弱，不利于凭证治理。
3. 调试/实验流量会干扰中心数据库性能。
4. 变更治理困难（schema 漂移风险更高）。

## 10. 实施步骤（建议）

1. 增加 `envId/nodeId` 字段到同步契约与分析模型。
2. 上线 Ingest API 与节点验签。
3. 在 Local/ECDS 各部署 Sync Agent（含离线队列）。
4. 上线冲突审计与补偿重放工具。
5. 建立节点治理台账（注册、吊销、配额、健康）。

## 11. 当前实现状态（2026-03）

1. `envId/nodeId` 已进入 `runs/sync` 与 ingest 写入链路，并做格式校验。
2. ingest 已支持单批次与批量模式（`{ batches: [...] }`）。
3. 节点验签已提供 HMAC 骨架与可开关强校验模式。
4. 失败补偿已覆盖自动重试、死信重投、run 级 replay 补齐。
