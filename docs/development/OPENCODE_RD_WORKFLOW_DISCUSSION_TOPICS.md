# OpenCode 研发流程讨论议题文档

## 文档目的

- 沉淀当前已达成方向与待讨论议题，作为后续方案评审输入。
- 本文档不作为最终定稿规范，结论以后续 plan/technical 文档为准。

## 当前已达成方向（摘要）

1. OpenCode 执行能力由 Agent Runtime 统一控制，支持状态感知与授权闭环。
2. 仅以下角色允许使用 OpenCode：
   - `engineering`
   - `operations`
   - `technical-expert`
3. 预算控制按 `agent + period`，超额度进入审批流。
4. 执行前必须做 Agent 绑定模型匹配检测。
5. `agent_runs` 扩展 `roleCode` 与 `executionData`。
6. 分析数据沉淀到 Engineering Intelligence，采用事件明细（A 方案）。
7. 多环境协同采用“中心控制面 + 边缘执行面 + Ingest 同步”，不采用各环境直连同库。

## 待讨论议题（优先级）

### P0（需先定）

1. **研发完成定义**
   - 是否以“任务执行完成”作为完成，还是加入额外验收门禁（测试/文档/审批）。
2. **讨论到执行的闭环触发**
   - 多 Agent 讨论后，何时自动生成任务清单，何时要求人工确认。
3. **超额度审批策略细化**
   - 不同角色、不同风险动作的审批人范围与时效（TTL）如何配置。

### P1（重要）

1. **模型回退策略**
   - 绑定模型不可用时，是否允许 fallback，允许到哪些模型。
2. **多环境冲突处理**
   - 同一 run 在 local/ecds 出现重复终态时的仲裁流程与告警策略。
3. **指标口径版本管理**
   - 成本/效率/惊喜度公式更新后的重算策略与版本追踪。

### P2（后续）

1. **GitHub 在研状态纳入**
   - 暂不纳入当前阶段，后续专题讨论任务状态与 PR/CI 事件联动。
2. **节点治理台账可视化**
   - local/ecds 节点注册、健康、凭证轮换、吊销页面设计。

## 会议建议模板（用于后续讨论）

每次讨论建议产出以下结构化记录：

1. `Decision`：本次明确结论（可执行）。
2. `Open Questions`：未决问题与依赖。
3. `Action Items`：责任人、截止时间、验收标准。
4. `Impact`：影响模块（agents / EI / frontend / ops）。

## 下一步建议

1. 先召开一次“研发完成定义 + 审批策略”评审会（P0）。
2. 评审后更新主计划：`docs/plan/OPENCODE_SERVE_INTERACTION_MASTER_PLAN.md`。
3. 将决策同步到技术设计与 API 文档，避免口径分叉。
