# 计划编排页面交互优化开发总结

## 1. 目标与范围

- 将计划编排页调整为“列表优先”的信息架构，降低进入页面后的操作成本。
- 将计划创建流程收敛为弹窗，减少页面主布局干扰。
- 将计划详情迁移到右侧抽屉，在不离开列表上下文的前提下完成查看与操作。

## 2. 实现内容

### 2.1 默认展示 Plan 列表

- 将页面主体改为列表首屏展示，进入页面即可浏览计划，不再以内联创建表单占据主区域。
- 列表列信息包含：计划标题/状态/模式/进度/更新时间，并提供详情入口。

对应文件：
- `frontend/src/pages/Orchestration.tsx`

### 2.2 创建计划弹窗化

- 新增创建弹窗状态 `isCreateModalOpen`，将原有创建表单迁移到 Modal。
- 保留原有创建字段（标题、prompt、mode、planner、autoRun）与校验逻辑。
- 创建成功后自动关闭弹窗、刷新列表、并打开新计划详情抽屉。

对应文件：
- `frontend/src/pages/Orchestration.tsx`

### 2.3 详情右侧抽屉化

- 新增详情抽屉状态 `isDetailDrawerOpen`，从列表行“查看详情”打开。
- 抽屉中承载计划基础信息、统计、任务列表、运行/删除/复制到新建等操作。
- 详情查询改为仅在抽屉打开时启用，减少无效轮询与请求。

对应文件：
- `frontend/src/pages/Orchestration.tsx`

### 2.4 状态一致性修正

- 删除计划后补充边界处理：当无剩余计划时自动关闭详情抽屉。
- 删除后刷新详情查询 key 指向下一条计划，避免使用旧 `selectedPlanId` 造成状态错位。
- 增加 `planTasks` 兜底，修复严格 TS 下 `tasks` 可空导致的编译报错。

对应文件：
- `frontend/src/pages/Orchestration.tsx`

## 4. 验证结果

- 前端构建通过：`npm run build:frontend`

## 5. 效果与收益

- 页面入口更聚焦：用户先看列表，再按需创建或查看详情。
- 交互链路更连续：列表、创建、详情三者通过 Modal/Drawer 解耦，切换成本更低。
- 数据请求更节制：详情查询与轮询仅在抽屉打开时触发。

## 6. 后续建议

- 可将列表筛选（按状态/模式）与排序（更新时间）做成可配置，提高历史计划检索效率。
- 可在详情抽屉引入任务分组（状态分组/执行者分组），进一步提升大计划可读性。
