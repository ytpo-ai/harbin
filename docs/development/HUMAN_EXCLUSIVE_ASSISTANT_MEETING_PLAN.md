## 背景

本次开发围绕“人类专属助理”机制展开，目标是让人类员工/高管在会议场景中通过其专属助理完成主持与发言，并在系统层面强制执行绑定前置条件。

## 最终业务规则

1. 人类员工/高管必须绑定专属助理，否则不可发起或参与会议。
2. 人类登录后若未绑定专属助理，系统弹出阻断引导，支持一键“创建并绑定专属助理”。
3. 人类发起会议时，会议主持人自动切换为其专属助理（host 为 agent）。
4. 人类本人不加入会议参与者列表。
5. 人类在会中发送消息时，后端自动映射为其专属助理身份发言。
6. 专属助理不主动响应，仅在其对应人类显式 `@` 时响应。

## 关键实现

### 1) 员工与专属助理绑定能力

- 新增员工字段：`exclusiveAssistantAgentId`，并与历史 `aiProxyAgentId` 兼容联动。
- 新增接口：
  - `POST /employees/:id/exclusive-assistant`（绑定）
  - `GET /employees/:id/exclusive-assistant`（查询）
  - `POST /employees/:id/exclusive-assistant/auto-create`（一键创建并绑定）
- 新增唯一性校验：同一个助理 Agent 不能同时绑定多个不同人类账号。

涉及文件：
- `backend/src/shared/schemas/employee.schema.ts`
- `backend/src/modules/employees/employee.service.ts`
- `backend/src/modules/employees/employee.controller.ts`

### 2) 登录后阻断引导

- 在全局布局中加载当前用户对应员工信息。
- 若当前用户为人类且未绑定专属助理，展示阻断弹层。
- 点击“创建专属助理”后调用 auto-create 接口，成功后自动解除阻断。

涉及文件：
- `frontend/src/components/Layout.tsx`
- `frontend/src/services/employeeService.ts`

### 3) 会议主持人与参与者逻辑

- 创建会议时，若请求 host 为人类，后端自动重写 host 为其专属助理（`hostType=agent`）。
- 不再把人类发起者自动加入 `participants`。
- 会议 participant 扩展了专属助理身份字段：
  - `isExclusiveAssistant`
  - `assistantForEmployeeId`

涉及文件：
- `backend/src/modules/meetings/meeting.service.ts`
- `backend/src/shared/schemas/meeting.schema.ts`

### 4) 人类发言自动代理为助理发言

- `POST /meetings/:id/messages` 当 `senderType=employee` 时：
  - 先解析该员工专属助理
  - 自动改写消息 `senderId/senderType` 为助理
  - 写入代理标记 metadata：`isAIProxy=true`、`proxyForEmployeeId`
- 保留“由人类触发会后续响应”的语义，避免因 sender 改写导致自动响应链路失效。

涉及文件：
- `backend/src/modules/meetings/meeting.service.ts`

### 5) 前端会议页体验调整

- 创建会议弹窗不再允许手动选择主持人，改为提示“将由专属助理主持”。
- 人类消息输入区域提示“将以专属助理身份发言”。
- 当前用户专属助理发出的消息按用户侧样式展示。
- 专属助理名称在参与者展示与 @ 提示中按“某某的专属助理”显示。

涉及文件：
- `frontend/src/pages/Meetings.tsx`
- `frontend/src/services/meetingService.ts`

# 人类专属助理会议接入计划

> 状态：已并入统一主计划 `docs/plan/MEETING_CAPABILITY_MASTER_PLAN.md`，本文件保留历史细节。

## 需求目标

系统将引入“人类专属助理”机制：所有人类员工和高管必须绑定一个专属助理 Agent；未绑定账号禁止发起和参与会议。发起会议时，专属助理以该人类的助理身份进入会议，并且仅在该人类显式 `@` 时响应，不主动触发。

## 执行步骤

1. 梳理现有用户、Agent、会议参与者模型与权限链路，明确“人类账号 + 专属助理”的绑定关系与身份表示方式。
2. 扩展数据模型与接口，支持员工/高管账号的专属助理绑定与查询，并为未绑定场景提供统一校验能力。
3. 改造会议创建与参会权限校验：未绑定专属助理的账号禁止发起会议、禁止参与会议。
4. 改造会议入会流程：人类用户发起会议时，自动将其专属助理以“该人类的助理身份”加入参与者上下文。
5. 改造会议消息触发逻辑：专属助理默认不主动响应，仅在对应人类用户显式 `@` 时触发助理处理。
6. 补充并更新测试与文档，覆盖绑定前置校验、会议权限、入会身份与 `@` 触发行为。

## 关键影响点

- 后端：用户绑定关系、会议权限守卫、消息触发策略。
- API：绑定接口、会议创建/参与错误码与提示约定。
- 数据库：用户与专属助理关联字段及历史数据兼容。
- 前端：会议入口拦截提示、会议内 `@` 交互与身份展示。
- 测试：权限边界、身份注入、会中触发规则。
- 文档：README、API 文档、功能说明更新。

## 风险与依赖

- 若当前消息系统缺少稳定的 `@mention` 解析链路，需要补充事件提取与归属校验。
- 需处理历史账号未绑定与历史会议数据兼容策略，避免上线后直接阻断业务。
- 若存在多端入口，需统一前端提示文案与后端错误语义，降低行为不一致风险。

## 增补需求（登录后自动引导绑定）

### 目标

人类员工/高管登录系统后，立即检测是否已绑定专属助理；若未绑定，页面阻断并提示创建。点击“创建专属助理”按钮后，系统自动创建助理 Agent 并完成绑定。

### 增补执行步骤

1. 扩展后端员工服务，新增“一键创建并绑定专属助理”接口，内部完成 Agent 创建、唯一性校验与绑定落库。
2. 保持鉴权/当前用户查询链路可用，前端在登录后初始化阶段获取当前用户对应员工信息并判断绑定状态。
3. 在全局布局增加阻断式引导弹层：未绑定专属助理时显示说明与“创建专属助理”按钮。
4. 用户点击后调用后端一键接口，成功后刷新当前用户员工信息并解除阻断；失败时给出明确错误提示与重试能力。
5. 更新 API/README 文档，补充登录后绑定引导与一键创建能力描述。

## 增补需求（二）：人类发起会议时由专属助理担任主持人

### 目标

当人类用户发起会议时，会议创建身份自动切换为该人类的专属助理；会议主持人固定为专属助理而非人类本人。

### 增补执行步骤

1. 改造会议创建后端逻辑：若创建请求的 host 为人类员工，自动解析其专属助理并重写会议 host 为该助理。
2. 在创建时自动将发起的人类加入参会者列表（普通参与者），并保持其专属助理为主持人角色。
3. 保持专属助理响应规则不变：仅在人类显式 `@` 时响应，主持身份不改变其被动触发策略。
4. 改造前端创建会议弹窗：移除手动主持人选择，明确提示“将由专属助理主持”。
5. 更新 API/README 文档描述，并执行构建验证。

## 增补需求（三）：人类不入会，直接以专属助理身份发言

### 目标

人类无需作为会议参与者入会；在会议中输入消息时，后端自动将发言身份映射为其专属助理。

### 增补执行步骤

1. 调整会议创建逻辑：人类发起时不再自动加入会议 participants，仅保留专属助理主持身份。
2. 调整会议消息写入逻辑：当 sender 为人类员工时，自动映射到其专属助理 `senderId/senderType`。
3. 保留“人类触发响应”的业务语义，避免改为 agent 发送后丢失后续讨论触发链路。
4. 调整前端输入区权限判断与提示文案：允许人类直接发言，但文案明确“以专属助理身份发送”。
5. 更新 API/README 文档并执行构建验证。

## 增补需求（四）：人类代理发言后，@ 专属助理仍可触发回复

### 目标

当人类消息被后端映射为“专属助理代发”后，`@` 触发规则仍应按“真实人类发送者”判断，确保 `@自己的专属助理` 能正常响应。

### 增补执行步骤

1. 在会议响应触发逻辑中，优先从消息 metadata 提取 `proxyForEmployeeId` 作为真实发送者。
2. 将专属助理触发条件从 `senderType === employee` 改为“存在真实人类发送者 ID”。
3. 专属助理归属校验继续使用 `assistantForEmployeeId === 真实发送者ID`。
4. 保持常规 Agent 触发逻辑不变，避免引入新的响应噪音。
5. 完成后执行 backend 构建验证。

## 增补需求（五）：代理发言时不要提前排除被 @ 的专属助理

### 目标

当人类消息被映射为“专属助理代发”时，系统不应因为“发送者ID=专属助理ID”而在预筛阶段提前移除该专属助理；否则即使显式 `@` 也无法响应。

### 增补执行步骤

1. 调整响应候选构建：先拿到在场 agent，再拆分为常规 agent 与专属助理两个集合。
2. 仅对常规 agent 施加“排除当前 senderId”规则；专属助理集合不在预筛阶段被移除。
3. 在 `@` 场景下，继续通过 `assistantForEmployeeId === 真实发送者ID` 严格限定可响应的专属助理。
4. 未命中时保持现有提示语义，避免静默失败。
5. 完成后执行 backend 构建验证。

## 增补需求（六）：日志查询意图强制路由到日志 MCP 工具

### 目标

在会议中当人类询问“我做了哪些操作/操作日志”等意图时，不依赖模型自由发挥，优先由被 @ 的专属助理调用 `human_operation_log_mcp_list` 返回真实日志摘要，避免“无记录”幻觉回复。

### 增补执行步骤

1. 在会议服务新增日志意图识别（中英文关键词），命中后触发专属路由。
2. 当且仅当命中 `@` 自己的专属助理时，优先调用日志 MCP 工具并生成结构化摘要回复。
3. 工具调用失败时回退到原有 Agent 响应链路，保证会议不中断。
4. 保持常规模型管理意图与普通对话路径不变。
5. 完成后执行 backend 构建验证。
