**完成事项**

- 按 `OPENCODE_AGENT_TASK_SSE_WORKER_PLAN` 与 `AGENT_TASK_SSE_MULTI_SERVE_PLAN` 落地 Agent Task 异步执行链路：新增 `POST/GET/cancel/SSE` 四类接口（`/agents/tasks/*`）。

- 新增 Multi-Serve Router（`OPENCODE_SERVE_REGISTRY`）与会话粘性能力：支持按负载权重选路、并发计数、`serveId` 落库。
- 网关新增 SSE 透传能力：`/api/agents/tasks/:taskId/events` 走 stream 代理，扩展 SSE 专用超时并关闭缓冲。
- 前端新增 `useAgentTaskSse` Hook（重连、去重、续传、降级轮询）与 `AgentTaskRunner` 页面，接入创建任务、实时事件、取消任务全链路。
- 新增任务超时与重试策略：落地 `stepTimeoutMs/taskTimeoutMs` 与指数退避重试（含 jitter），并补充 `retry_scheduled/retry_started` 观测事件与任务查询字段（attempt/nextRetryAt）。
- API 文档更新：`docs/api/agents-api.md` 增补 Agent Task SSE 与 Multi-Serve 配置说明。
- 修复 OpenCode endpoint 命中优先级：统一为 `agent.config.execution.endpoint` > `agent.config.execution.endpointRef` > runtime endpoint > env default，避免误回退到 `OPENCODE_SERVER_URL`。
- 修复“任务成功但 Token 输出为空”：扩展 OpenCode message 响应解析（`info.content/content/parts/info.parts/payload.parts/message/output`），并在执行桥接中按事件 delta 实时透传 `token`。
- 新增运行提示文档：`docs/TIP.MD`，强调 OpenCode endpoint 强制优先级与排障要点；`AGENTS.md` 增加引用入口。
- 修复取消链路 endpoint 漂移：OpenCode 会话取消复用执行时 resolved endpoint，避免误打 env 默认端口。
- 取消接口收敛为 OpenCode Web 实测可用路径：`POST /session/:id/abort`（移除其他取消 fallback），并新增 `abort start/success/failed` 日志便于定位。
- 修复 Agent 执行入口 `POST /api/agents/:id/execute` 对业务标识（如 `executive-lead`）的兼容性：查询从仅 `_id` 扩展为 `_id/id` 双通道，避免 `Cast to ObjectId failed` 并统一返回业务语义错误。
- Orchestration 执行链路去同步阻塞：`orchestration.service` 改为提交 Agent Task 异步任务（`POST /agents/tasks`）并通过状态查询回传终态（`GET /agents/tasks/:taskId`），不再直接 `await executeTaskDetailed`。
- Orchestration 终态等待升级为事件驱动：优先订阅 Agent Task SSE（`GET /agents/tasks/:taskId/events`）消费终态事件，异常自动回退轮询，降低状态查询开销。
- Agent Task worker 完成态补充持久化 `resultSummary.response`，使编排侧可稳定回收输出用于校验与结果落库。
- 计划创建链路升级为异步流式：`POST /orchestration/plans/from-prompt` 改为先落库 `drafting` 并秒回 `planId`，后台逐条生成任务并落库。
- 新增计划 SSE 事件流接口：`GET /orchestration/plans/:id/events`，推送 `plan.status.changed`、`plan.task.generated`、`plan.completed`、`plan.failed`。
- 前端 Orchestration 创建成功后自动跳转计划详情页；详情页在 `drafting` 态展示“任务生成中”，并按事件实时追加任务（step-like 体验），断线时轮询兜底。

- 修复 Agent Task 取消流程端到端失效问题（4 个根因）：
  - `executeTaskWithStreaming` 未调用 `onStarted` 回调导致 `runId` 始终为空
  - `promptSession` HTTP 请求无法被中断（新增 `AbortController` + `AbortSignal` 机制）
  - `cancelSession` 未中断正在等待的 HTTP 请求（新增 `activeAbortControllers` Map）
  - 状态判定逻辑错误：有响应时 `cancelRequested=true` 仍标记为 `succeeded`（移除 `!hasResponse` 条件）
- 修复后取消响应时间从 30~45 秒降至 ~9 秒，且最终状态正确为 `cancelled`
- 第二轮修复 Agent Task 取消流程遗留问题（5 项）：
  - 取消响应 ~9s 延迟优化至毫秒级：`AgentTaskService` 新增 `EventEmitter cancelEmitter`，`cancelTask` 保存后立即 emit 事件；Worker `waitForTaskCancellation` 重写为 EventEmitter 事件驱动 + 轮询 fallback 双模式
  - `cancelRuntimeRun` reason 硬编码修复：新增可选 `reason` 参数，用户取消传 `'user_cancel'`，超时取消传 `'step_timeout_cancel'`
  - OpenCode abort 无重试修复：`cancelSession` 新增 3 次重试逻辑（间隔 1s、2s）
  - SSE 取消状态误导修复：SSE event `status` 从 `task.status`（`'running'`）改为 `'cancelling'`
  - `throwOnError` 无效修复：`request()` catch 块修正，`throwOnError=false` 时返回 `undefined` 而非重新抛出
- 修复 Runtime EI 同步 404：`RuntimeEiSyncService` 统一将 `ENGINEERING_INTELLIGENCE_SERVICE_URL` 规范化为包含 `/api` 前缀，并固定同步目标为 `/api/ei/sync-batches`，避免终态 run 重试至 dead-letter
- 收敛 legacy 客户端模块：将 `models-client` 与 `tools-client` 能力并入 `agents-client`，并删除旧模块文件与 `AppModule` 相关装配项。
