# 前端开发设计建议规范

> 本规范定义了项目前端开发中的设计建议，供开发人员在进行前端开发时参考遵循。

## 1. 技术栈

- **框架**: React + TypeScript
- **样式**: Tailwind CSS
- **组件库**: 基于 Ant Design 定制
- **状态管理**: React Hooks / Context
- **请求**: 自定义 service 层封装

## 2. 布局规范

### 2.1 页面结构

- **列表优先**: 页面默认展示列表视图，用户进入页面默认可浏览和选择
- **创建弹窗化**: 创建操作收敛为弹窗流程，减少页面跳转和上下文切换
- **详情抽屉化**: 详情/编辑操作使用右侧抽屉承载，保持上下文连续性

### 2.2 布局原则

- 避免多级页面跳转，优先使用弹窗/抽屉
- 弹窗承载简单表单和确认操作
- 抽屉承载复杂编辑、多 Tab 切换、详情查看
- 避免页面内多区块平行布局产生的状态耦合

## 3. 组件规范

### 3.1 Tab 使用场景

- 同一区域需要承载多个子功能时使用 Tab 切换
- 推荐模式：`执行/修改` Tab、`详情/Agent 绑定` Tab、`调试/Session` Tab
- Tab 切换可减少弹窗切换成本，提升操作效率

### 3.2 列表设计

- 列表项高度限制 2-3 行，超出部分渐隐处理
- 标题/状态/操作统一放入 item header 栏
- 复杂详情通过「查看详情」进入右侧抽屉
- 减少列表噪音，突出核心识别信息

### 3.3 卡片设计

- Agent/人物类列表使用人物卡片样式展示
- 头像为空或加载失败时展示默认头像
- 卡片内保留核心信息与操作入口

## 4. 交互规范

### 4.1 状态管理

- **乐观更新**: 保存操作使用乐观更新，失败时回滚并显示错误提示
- **保存中态**: 提交过程中显示 loading 状态，避免重复提交
- **状态同步**: 跨组件状态通过事件广播保持同步（如 `message-center:updated`）

### 4.2 操作收敛

- 合并相似操作语义，移除重复入口
- 示例: 将「检索」与「重建」收敛为单图标菜单入口
- 示例: 将「执行」与「编辑」合并为抽屉 Tab

### 4.3 交互反馈

- 操作前提供二次确认（如删除/弃用操作）
- 错误状态提供可读的错误提示和重试路径
- 空状态、加载态、错误态保持视觉一致性

## 5. 视觉规范

### 5.1 信息层级

- 主信息（状态标签、动作语义）突出展示
- 辅助信息（时间、来源、上下文）次要展示
- 关键信息使用标签/徽章等视觉强化

### 5.2 文案中文化

- 状态、分类、筛选项统一使用中文展示
- 避免中英混杂，保持文案一致性

### 5.3 字段对齐

- 前端模型与后端 API 字段完全对齐
- 统一字段名、类型、枚举值、默认值
- 避免前后端字段映射偏差导致的回显异常

### 5.4 空值兜底

- 头像为空时显示默认头像
- 列表为空时显示空态提示
- 字段缺失时提供合理的默认值或隐藏

## 6. 性能与质量

### 6.1 列表优化

- 列表链路绝不加载正文内容
- 分页加载避免一次性请求大量数据
- 长列表考虑虚拟滚动

### 6.2 可访问性

- 焦点管理：弹窗/抽屉打开时聚焦内部，关闭时恢复
- 键盘操作：支持 ESC 关闭弹窗/抽屉
- 屏幕阅读：保证语义化标签和 aria 属性

### 6.3 响应式适配

- 确保桌面端和移动端均可正常阅读和操作
- 窄屏环境下避免信息过度密集

## 7. 代码规范

### 7.1 命名规范

- 组件: PascalCase（如 `AgentList.tsx`）
- 文件: kebab-case（如 `agent-list.tsx`）
- 函数/变量: camelCase
- 常量: UPPER_SNAKE_CASE

### 7.2 组件结构

- 使用函数式组件 + React Hooks
- 逻辑复用优先使用自定义 Hook
- 避免业务逻辑直接写在组件内

### 7.3 Service 层

- API 调用封装在 service 层
- 统一错误处理和 loading 状态管理
- 类型定义遵循后端 DTO 命名

## 8. 文档维护

- 新增页面需更新 `docs/feature/` 相关功能文档
- API 变更需更新 `docs/api/` 接口文档
- 交互流程变更需更新使用文档
