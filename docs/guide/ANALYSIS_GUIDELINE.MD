# 编排计划执行分析指南（踩坑记录）


## 一、走过的弯路

### 弯路 1：用过期的静态 JSON 快照代替数据库查询

**错误做法**：首次分析时直接读取了 `data/agents.json`（2026-03-09 快照），拿半个月前的数据来判断 agent 的工具配置。

**导致的错误结论**：
- 基于 JSON 快照认为"系统中没有任何 agent 同时拥有 repo-read + repo-writer + save-prompt-template"
- 实际查数据库后发现 **Docter-W 和 Coder-T 都完全匹配**，它们是 JSON 快照之后才创建/配置的 agent

**正确做法**：涉及运行时状态的分析，**必须先查数据库**。静态文件（JSON、seed 脚本）只能作为"设计意图"参考，不能代替"运行时实际状态"。

### 弯路 2：依赖代码推断代替事实查证

**错误做法**：通过阅读 `mcp-profile.ts` 的 seed 配置来推断每个 agent 的工具列表，然后得出"没有 agent 具备完整工具"的结论。

**导致的错误结论**：
- seed 脚本定义的是角色级别的工具白名单模板，不是 agent 实例的实际配置
- agent 的 `tools[]` 可以在运行时通过 API/管理界面被单独修改，与 seed 脱节
- 例如 Docter-W 的 `role-technical-architect` seed 中没有 `repo-writer` 和 `save-template`，但数据库中该 agent 实际拥有这些工具

**正确做法**：seed 脚本 = 初始模板，数据库 = 运行时真相。分析运行时问题时以数据库为准。

### 弯路 3：在没有证据时就下归因结论

**错误做法**：第一轮分析中直接声称"Planner 未调用 list-agents 做工具预校验"，将其作为 Planner 的过失。

**被质疑后发现**：
- 系统架构设计中 `buildAgentManifest()` 已经在代码层自动将完整 agent+tools 清单注入 Planner prompt，Planner 不需要额外调用 `list-agents` 工具
- "是否调用了 list-agents"这个判断需要查执行日志或工具调用记录，不能凭代码逻辑猜测

**正确做法**：归因前先固定事实。对"某人做了/没做某事"的判断，必须有日志、数据库记录或代码执行路径的直接证据，而不是基于"应该做但可能没做"的推测。

### 弯路 4：先写结论再找证据（确认偏误）

**错误做法**：第一轮分析的思路是"先列出问题清单 → 给每个问题归因"，实质上是先有了"Planner 有问题、Agent 有问题"的框架，然后选择性地找支持这些结论的证据。

**具体表现**：
- 把 Planner 没调用 list-agents 当成"盲派"证据，但实际系统自动注入了 manifest
- 把"系统中无匹配 agent"当成环境问题，但实际数据库中 Docter-W 完全匹配
- 这些错误结论互相支撑，形成了一个看起来自洽但基于错误前提的分析

**正确做法**：先穷举事实（数据库状态、执行日志、代码路径），再从事实中归纳结论。遇到与预设不符的事实时，修改结论而不是忽略事实。

### 弯路 5：忽略了时间维度的数据变化

**错误做法**：把 `data/agents.json`、`mcp-profile.ts` seed、数据库当成同一时刻的快照混用。

**实际情况**：
- `data/agents.json` 是 2026-03-09 导出的，只有 6 个 agent
- 数据库中有 13 个 active agent（Docter-W 是 `69bc` 开头的 ObjectId，创建时间晚于 03-09）
- 计划执行时间是 2026-03-23/24，中间 agent 团队已经扩充

**正确做法**：分析编排计划执行问题时，必须以**计划执行时刻的数据库状态**为基准。如果需要参考静态文件，要先确认其时间戳是否覆盖计划执行窗口。

## 二、正确的分析流程（应该怎么做）

### Step 1：从数据库获取运行时事实

执行分析前，先查以下数据：

```
1. orchestration_plans  → 计划元信息、状态、taskIds
2. orchestration_tasks  → 每个任务的 assignment、status、runLogs、runtimeTaskType、error
3. agents              → 所有 active agent 的 name、tools[]、roleId、tier（运行时实际值）
4. agent_tools         → 关键工具的注册状态（是否 enabled）
```

### Step 2：构建工具匹配矩阵

将任务所需工具与所有 agent 的实际工具做交叉比对，明确回答：
- 系统中是否存在完全匹配的 agent？
- Planner 自己匹配几项？被分配者匹配几项？
- 最佳候选是谁？

### Step 3：逐任务核对分配决策

对每个任务检查：
- Planner 指定的 agentId 是谁？
- 是 Planner 直接指定还是系统 fallback？（看 `fallbackUsed` 字段）
- 被分配者是否具备该任务所需工具？
- 如果不具备，系统是否有拦截机制？（查代码中的校验路径）

### Step 4：区分"信息不足"和"决策失误"

- 查 `buildAgentManifest()` 注入的上下文，确认 Planner 在规划时是否能看到所有 agent 的工具信息
- 如果能看到但仍选错 → LLM 决策问题
- 如果看不到（manifest 被截断、agent 未加载等）→ 系统上下文构建问题

### Step 5：基于事实归因，不做推测

每条结论必须附带数据来源（数据库字段值、代码行号、日志内容），不接受"应该是"、"可能是"、"大概率是"的措辞。

## 三、核心原则

1. **数据库优先**：运行时问题看数据库，不看静态文件
2. **证据先行**：归因前先固定事实，不做无证据推测
3. **时间对齐**：确认数据来源的时间戳是否覆盖分析目标的执行窗口
4. **区分层次**：seed 配置 != 数据库状态 != 代码逻辑 != 运行时行为，每层都可能有差异
5. **反确认偏误**：主动寻找反驳当前结论的证据，而不是只找支持的
