# CTO 研发流程 Skill（附录版）

本文件是 `docs/skill/cto-rd-workflow.md` 的补充材料，按需加载。

## A. 触发词与触发判定规则

### A.1 高概率触发词

出现以下语义信号时，优先考虑启用核心 skill：

- `做一个需求`
- `实现一个功能`
- `新增一个能力`
- `优化一下这个功能`
- `修一下这个问题`
- `排查这个报错`
- `补一份文档`
- `出一个方案`
- `做个技术方案`
- `给这个需求排计划`
- `推进开发`
- `提 PR / 提交代码`

说明：不要求逐字匹配，语义接近即可。

### A.2 推荐触发场景

- 用户希望需求进入研发处理，而非停留讨论
- 需要先理解和分类，再进入方案或开发
- 事项将涉及研发链路中两个及以上阶段
- 需要 CTO 先分诊编排，再决定 agent 分配
- 事项需要状态跟踪与回退机制

### A.3 不触发场景

- 纯知识问答或概念解释
- 纯信息查询
- 一次性小修改，不需要流程化协作
- 用户明确仅讨论，不进入研发流程
- `step0` 外部受理动作

### A.4 最小判定规则

建议触发核心 skill 的条件：

1. 当前事项属于 `feature` / `fix` / `doc`
2. 当前事项需要进入研发链路中的两个及以上阶段

若只满足第 1 条：可先按普通协助处理。

若第 1 条暂无法判断：先做简短澄清，再决定触发。

### A.5 快速判断问句

- 这是不是要推进的研发事项，而非单纯讨论？
- 是否需要先做需求理解或分类？
- 在方案完成前，是否不适合直接派开发？
- 是否需要状态跟踪或人类兜底？

多数为“是”时，触发核心 skill。

### A.6 触发后的默认动作

1. 进入 `step1`：整理理解与未决问题
2. 进入 `step2`：输出主类型标签
3. 有关键阻塞则转 `pending-meeting`
4. 无关键阻塞则推进 `step3`
