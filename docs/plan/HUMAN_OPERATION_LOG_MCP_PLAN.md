# 人类系统操作日志与 MCP 工具建设计划

## 需求理解

- 需要将人类在系统中的关键操作记录到数据库，形成可审计日志。
- 需要新增日志 MCP 工具，供“人类专属助理”查询操作轨迹并辅助理解上下文。
- 专属助理必须被严格限制：仅可查看其绑定人类的操作日志，禁止跨人类访问。

## 执行步骤

1. 梳理“系统操作”覆盖范围，定义统一日志模型（主体、动作、资源、结果、上下文、时间）与脱敏规则。
2. 新增操作日志数据模型与索引（重点覆盖 `humanEmployeeId + timestamp`、`assistantAgentId + timestamp`、`action`），确保高效查询与审计回溯。
3. 在后端请求链路接入日志采集（优先通过 gateway 代理链路记录），将人类身份、请求信息、执行结果写入日志库。
4. 在 agents 服务实现日志查询服务，内置“专属助理 -> 绑定人类”校验逻辑，只返回绑定人类日志。
5. 新增日志 MCP 工具（如 `human_operation_log_mcp_list`），支持时间区间、动作类型、资源关键词、分页查询。
6. 补充测试与文档：覆盖日志写入、权限隔离、MCP 过滤行为，并更新 API/README 说明。

## 关键影响点

- 后端：gateway 请求转发链路与 agents 工具执行模块。
- 数据库：新增操作日志集合与查询索引。
- 权限安全：专属助理绑定关系校验、跨人类访问拦截、敏感字段脱敏。
- API/MCP：新增日志查询工具能力与响应结构。
- 文档：`README.md`、`docs/api/API.md` 与相关说明更新。

## 风险与依赖

- 日志量增长快，必须限制默认查询窗口并配合索引/分页。
- 请求体可能包含敏感字段，需统一脱敏后再落库。
- 绑定关系依赖员工数据质量（`exclusiveAssistantAgentId`），历史脏数据可能导致权限误判。

## 验证方式

- 功能验证：人类执行操作后可生成日志；专属助理可查询绑定人类日志；非绑定助理无法读取。
- 安全验证：日志中敏感字段（password/token/secret 等）被脱敏。
- 质量检查：通过 backend lint、typecheck、test 与构建验证。

## 增补范围（前端日志查询页 + 人类全员可见）

1. 在 legacy 后端新增日志查询 API（分页 + 多条件筛选），供前端直接调用。
2. 调整该 API 的访问策略为“任意已登录人类用户可查看全员日志”（保持日志字段脱敏策略不变）。
3. 前端新增“日志查询”页面，支持时间范围、动作/资源关键词、成功状态、状态码、人类ID筛选与分页浏览。
4. 在导航与路由中挂载页面入口，确保登录用户可直接访问。
5. 补充 API/README 文档，明确“前端日志查询能力与可见性范围”。

### 增补影响点

- 后端：新增 operation logs 查询模块与鉴权校验。
- 前端：新增页面、服务与路由/导航入口。
- 权限策略：从“专属助理仅看绑定人类”扩展为“人类用户可看全员日志（前端入口）”。

## 增补范围（日志查询接口不回写）

1. 在 gateway 日志采集流程增加排除规则：日志查询接口（`/api/operation-logs`）调用不写入 `operation_logs`。
2. 保持其他接口日志采集能力不变（脱敏、状态码、耗时、来源服务等）。

### 增补影响点

- 后端：`gateway-proxy.service` 的日志写入条件判断。
