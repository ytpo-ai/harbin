# Agent Remove Type Keep Role Plan

## 1. 背景与目标

- 当前 Agent 同时维护 `type` 与 `roleId`，在语义与默认配置上有重复。
- 目标是以 `roleId`（及对应 `role.code`）作为唯一业务身份来源，移除前端“类型”概念。
- 兼容历史存量数据：先完成“行为去 type”，再执行“数据去 type”。

## 2. 范围

### 2.1 In Scope

- Agent 创建/编辑流程移除 type 输入，仅保留角色。
- 后端创建/更新不再强依赖 `type` 必填。
- MCP 配置、工具白名单、展示文案优先基于角色。
- 受影响页面统一从“类型”展示切换为“角色”展示。

### 2.2 Out of Scope

- 本阶段不做破坏性数据库字段删除。
- 本阶段不强制重写全部历史记录中的 `type` 值。

## 3. 执行步骤

1. **后端契约兼容化（Role First）**
   - 创建/更新 Agent 时 `type` 改为可选。
   - 关键流程（工具白名单、MCP profile 映射）优先使用 `role.code`，`type` 仅兜底。
2. **前端表单去类型化**
   - Agent 创建/编辑去掉“类型”字段与校验。
   - 角色选择后自动使用 `role.promptTemplate` 作为默认 prompt（仅在未手改或仍为旧默认时覆盖）。
3. **展示层统一角色语义**
   - Agent 列表、详情、讨论、会议等页面将“类型”显示改为“角色”。
4. **角色初始化文案对齐**
   - HR 页面中“从 agent_type 初始化”文案改为系统角色模板语义，避免概念冲突。
5. **回归验证**
   - 验证 Agent 创建/编辑、工具白名单、生效的 MCP 能力与会议相关逻辑。
6. **二阶段清理准备**
   - 记录仍依赖 `type` 的边界点，为后续物理删字段提供清单。

## 4. 关键影响点

- 后端/API：Agent 创建更新契约、MCP profile 匹配逻辑。
- 前端：Agent 管理与多处展示页面。
- 数据：历史 Agent 记录继续保留 `type`，但不再作为主驱动。
- 文档：相关功能文档需更新“角色优先”的设计说明。

## 5. 风险与应对

- 风险：部分隐式逻辑仍读取 `agent.type`（如系统内置 Agent 识别）。
  - 应对：保留兜底判断并逐步迁移到 role code。
- 风险：旧默认 Prompt 与新角色模板覆盖策略不一致。
  - 应对：仅在“未手动编辑”条件下自动覆盖。
- 风险：外部调用方仍传 `type`。
  - 应对：API 继续兼容接收，不再要求必填。

## 6. 验收标准

- 前端 Agent 创建/编辑不再出现“类型”输入项。
- 仅通过角色即可成功创建和更新 Agent。
- 工具权限与 MCP profile 在无 `type` 输入场景下仍正确生效。
- 主要页面对 Agent 的身份展示统一为角色。
