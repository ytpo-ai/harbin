# 统一消息表（messages）实施计划

## 需求目标

将系统中的会议、任务编排等消息统一沉淀到 `messages` 集合，建立统一可检索、可评测、可统计的消息基座，用于模型输出评估与 Agent 绩效分析。

## 执行步骤

1. 定义统一消息模型与索引：新增 `messages` schema，覆盖场景标识、发送者信息、内容、元数据与评测字段。
2. 建立统一消息服务：封装写入、分页查询、最近消息查询能力，避免业务模块重复实现。
3. 接入会议消息链路：会议用户消息/系统消息统一写入 `messages`，保留现有行为兼容。
4. 接入任务编排会话链路：orchestration session 的对话消息统一写入 `messages`，补齐任务上下文维度。
5. 更新文档与验证：补充 API/架构说明，完成基础回归验证（创建会议、发言、任务会话写入）。

## 关键影响点

- 后端：`meetings`、`orchestration` 模块服务改造与依赖注入。
- 数据库：新增 `messages` 集合与复合索引。
- API：新增统一消息查询接口（按场景分页）。
- 评测与绩效：统一数据口径，可按模型/Agent/场景聚合统计。

## 风险与依赖

- 兼容性风险：前端和历史逻辑仍依赖原嵌套消息字段，需先双写兼容再逐步收敛。
- 数据规模风险：统一表写入量提升，必须提前建立关键索引。
- 口径风险：不同模块消息类型不一致，需通过 `sceneType/messageType/metadata` 规范化。
