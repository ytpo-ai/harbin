# TODO/History Memo 内容与聚合优化计划

## 需求背景

当前 memo 设计中，`todo` 与 `history` 的职责边界不够清晰，容易混入会议聊天内容，且“未执行任务”和“已执行任务状态”在聚合口径上存在重叠。

本次优化目标：

1. `todo` 只记录任务编排中的未执行任务。
2. `todo` 不记录会议聊天内容。
3. `history` 记录已执行的编排任务及其状态。

## 执行步骤

1. 梳理 `todo` / `history` 的领域边界，明确“编排任务”和“会议聊天”的来源划分。
2. 定义统一任务状态机与归档口径：未执行任务归入 `todo`，进入执行态及终态任务归入 `history`。
3. 设计聚合模型（事件流 + 视图）：以任务 ID 为主键进行幂等聚合，支持状态迁移时自动出入队。
4. 输出写入过滤规则：`todo` 仅接收 orchestration task 事件，显式拒绝 meeting chat 事件。
5. 形成文档化方案：更新功能文档，并补充技术设计文档（数据结构、聚合流程、一致性与回溯机制）。
6. 定义验收标准与测试场景（状态迁移、并发写入、去重、回放重建、聊天过滤）。

## 关键影响点

- 后端：memo 写入链路、状态聚合器、任务状态映射。
- API：`todo` / `history` 查询口径与字段约束。
- 前端：两个备忘录视图的数据展示语义。
- 数据层：索引、去重、幂等键、重建能力。
- 测试：状态流转正确性与过滤规则回归。
- 文档：功能文档与技术文档同步更新。

## 风险与依赖

- 风险：状态命名不统一导致任务归类偏差。
- 风险：历史全量事件保留与查询性能之间的平衡。
- 依赖：编排任务事件需要稳定的任务 ID、事件时间戳与状态字段。
