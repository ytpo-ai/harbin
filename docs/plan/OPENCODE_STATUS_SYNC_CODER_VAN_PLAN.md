# OpenCode 状态同步排障（coder-van）计划

## 1. 需求理解

- 问题现象：研发 Agent 执行 OpenCode 时，状态未按预期同步到平台侧。
- 目标：基于 `coder-van` 账号完成链路复现与定位，明确根因并给出最小修复方案。

## 2. 执行步骤

1. 明确状态链路与观测点：梳理 Runtime 事件、run 状态、EI 同步状态、前端状态来源。
2. 使用 `coder-van` 身份最小复现一次执行，并记录 runId、sessionId、关键时间点。
3. 对比状态在各层的一致性：Runtime API / Orchestration 任务状态 / EI 同步状态。
4. 排查关键节点：事件发布（Hook/outbox）、状态持久化、消费推送（WS/SSE/轮询）。
5. 若发现缺陷，实施最小变更修复并补充必要测试或校验脚本。
6. 回归验证：再次用 `coder-van` 复测，确认状态同步闭环恢复。
7. 形成结论：输出根因、影响范围、修复点与后续防回归建议。

## 3. 关键影响点

- 后端 Runtime（run 生命周期、hook/outbox）
- 后端 Orchestration（任务状态回写与计划执行状态）
- EI 同步链路（run 终态同步与重试状态）
- 前端状态消费（run 状态查询与实时订阅）

## 4. 风险与依赖

- Token 时效性可能影响复现窗口。
- 若存在并发写入或补偿重试，可能出现短时状态抖动，需要结合时间线判断。
- 若依赖本地 Redis/Mongo/各服务进程状态，需先确认服务健康。
