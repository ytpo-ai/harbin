# Agent 日志「决策回溯」抽屉 Plan

## 背景

扣分记忆（deduction memo）功能已实现并注入 LLM 上下文，但扣分重复率未见明显下降（如 generating 阶段仍然 77 分、initialize 64 分）。当前存在两个核心问题：

1. **不可观测**：无法确认某次任务执行时 LLM 实际收到了哪些 system prompt、deduction context 是否在其中
2. **不可追问**：任务结束后无法回溯 LLM 为什么做出错误行为，哪些上下文因素影响了它的输出

## 目标

在 Agent 详情页的 **LogTab** 中，为每个 Run 卡片增加「决策回溯」入口按钮，点击后打开一个独立抽屉，提供：

1. **Prompt 审查**：直接查看 LLM 收到的完整 system messages（含 deduction block），验证扣分记忆是否真正到达
2. **执行回放**：逐轮查看 LLM 的输入→输出→工具调用→结果链路，并关联每轮的扣分
3. **LLM 自诊断**：一键让 LLM 根据完整执行记录分析自己为什么做出错误决策

## 为什么放在 Agent 日志页

| 考量 | 说明 |
|------|------|
| 数据直接可用 | LogTab 已有 `loadRunMessages(runId)` + `loadRunScore(runId)`，`runId` 直接可取 |
| 不需要修后端 | 不涉及 OrchestrationRunTask 缺少 `agentRunId` 的链路断裂问题 |
| 已有 UI 基础 | LogTab 已有 flow/raw/score 三个 Tab，评分类型、规则常量、badge 样式全部就绪 |
| 用户路径自然 | 从 Agent 日志看到 77 分 → 直接点进去回溯，比从编排任务详情绕更直接 |

## 设计方案

### 1. 入口

在 LogTab 每个 Run 卡片的**展开区域**内，现有 `flow | raw | score` Tab 栏右侧，新增一个「决策回溯」按钮。

```
┌─────────────────────────────────────────────────────────────┐
│ 根据 rd-workflow 技能执行需求开发 -20 任务#2     成功  77分  │
│ generating · 耗时: 23.2s · Agent: Kim-CTO                   │
├─────────────────────────────────────────────────────────────┤
│  [执行流程] [原始信息] [扣分记录]              [决策回溯 →]  │
│  ┌─ 当前 Tab 内容 ─────────────────────────────────────┐    │
│  │ ...                                                 │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

- 按钮条件：`run.status` 为 `completed` 或 `failed`
- 按钮样式：`text-xs font-medium text-primary-600 hover:text-primary-700`，与现有「查看 Session →」风格一致

### 2. 决策回溯抽屉

点击后打开右侧全屏抽屉（z-[48]，在 Agent 详情 Session 抽屉 z-40 之上）。

```

│  Agent: Kim-CTO  ·  77/100分  ·  耗时: 23.2s  ·  7 轮          │
├─────┬────────────────────────────────────────────────────────────┤
│     │  [Prompt 审查]  [执行时间线]  [诊断分析]                    │
│     ├────────────────────────────────────────────────────────────┤
│ 轮  │                                                            │
│ 次  │                                                            │
│ 导  │              Tab 对应的主内容区                              │
│ 航  │                                                            │
│     │                                                            │
│ R1  │                                                            │
│ R2⚠ │                                                            │
│ R3  │                                                            │
│ R4  │                                                            │
│ R5⚠ │                                                            │
│ R6  │                                                            │
│ R7  │                                                            │
└─────┴────────────────────────────────────────────────────────────┘
```

**抽屉宽度**：`w-full sm:w-[92vw] lg:w-[72vw]`（比现有 DebugDrawer 的 56vw 更宽，因为内容更丰富）

### 3. 头部信息栏

展示当前 Run 的关键指标，一行内紧凑排列：

| 指标 | 数据来源 |
|------|---------|
| 任务标题 | `run.taskTitle` |
| Agent 名称 | `run.agentName` |
| 评分 | `scoreData.score` + badge 颜色 |
| 耗时 | `totalDurationMs` 格式化 |
| 轮次数 | `scoreData.stats.totalRounds` |
| 总扣分 | `scoreData.totalDeductions` |
| 工具成功率 | `successfulToolCalls / totalToolCalls` |

### 4. 侧边栏：轮次导航

垂直时间线，固定宽度 `w-14`。

```
 R1 ●──
    │
 R2 ●── ⚠️
    │
 R3 ●──
    │
 R4 ●──
    │
 R5 ●── ⚠️
```

- 每个节点是一个圆点 + Round 编号
- 颜色规则：
  - 绿色（`bg-emerald-500`）：该轮无扣分、无错误
  - 橙色（`bg-amber-500`）：该轮有扣分
  - 红色（`bg-rose-500`）：该轮工具执行失败
- 点击某轮：
  - 如果当前在「执行时间线」Tab，滚动到对应 Round
  - 如果当前在其他 Tab，先切换到「执行时间线」再滚动
- 当前激活轮：圆点放大 + ring 高亮

### 5. Tab A：Prompt 审查

**核心价值**：一眼确认 deduction context 是否出现在 LLM 的 system messages 中。

**数据来源**：`runMessages` 中筛选 `role === 'system'` 的消息（已在 `useLogState` 中加载）

**UI 结构**：

```
┌──────────────────────────────────────────────────────────┐
│  System Messages (共 N 条)                               │
│                                                          │
│  ┌─ #1  Identity ─────────────────────────────────────┐  │
│  │ ▶ 你是 Kim-CTO，负责...                [展开全文]  │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌─ #2  Toolset ──────────────────────────────────────┐  │
│  │ ▶ 可用工具列表：git_pull, read_file... [展开全文]  │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌─ #3  ⚠️ 扣分记忆 ──── border-l-4 amber ───────────┐  │
│  │ ▼ 【执行质量提醒】                                  │  │
│  │   ## 高频错误与改进要求                              │  │
│  │   1. D3 连续调用相同工具 (8次, -80分)               │  │
│  │   2. D11 达到最大轮次上限 (3次, -45分)              │  │
│  │   ...                                               │  │
│  │                                                     │  │
│  │   ## 近期具体错误案例                               │  │
│  │   ...                                               │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌─ #4  Task ─────────────────────────────────────────┐  │
│  │ ▶ 任务描述：根据 rd-workflow...        [展开全文]  │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌─ #5  Memory ───────────────────────────────────────┐  │
│  │ ▶ Session 历史摘要...                  [展开全文]  │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌─ ❌ 异常检测 ──────────────────────────────────────┐  │
│  │ ⚠ 本次执行未检测到扣分记忆注入                     │  │
│  │   可能原因：Redis 缓存为空 / 聚合未触发             │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

**关键设计**：
- **自动标签识别**：根据 system message 的 content 关键字自动识别 context layer 类型：
  - 包含 `执行质量提醒` → 标记为「扣分记忆」
  - 包含 `你是` / `You are` → 标记为「Identity」
  - 包含 `可用工具` / `Available tools` / `tool_call` 格式说明 → 标记为「Toolset」
  - 包含任务描述文本 → 标记为「Task」
  - 其他 → 标记为「System #N」
- **扣分记忆卡片**特殊处理：
  - 左侧边框 `border-l-4 border-amber-400`
  - 默认展开（其他默认折叠）
  - 内容区使用暗色代码块：`bg-slate-900 text-slate-100 font-mono text-xs whitespace-pre-wrap rounded-lg p-4`
- **缺失检测**：如果所有 system messages 中都不包含 `执行质量提醒`，底部显示红色警告卡片
- 折叠态预览前 2 行内容，`line-clamp-2 text-slate-500`

### 6. Tab B：执行时间线

**核心价值**：逐轮回放 LLM 交互过程，关联每轮扣分。

**数据来源**：
- `runMessages` 中 `role !== 'system'` 的消息（排除初始 system messages），按 `stepIndex`（如果有）或消息顺序分组
- `scoreData.deductions` 按 `round` 关联到对应 Round

**分组逻辑**：
1. 按 `stepIndex` 字段分组（如果 messages 有该字段）
2. 如果没有 stepIndex，按消息顺序推断：每个 `assistant` message 及其后续的 `tool` messages 为一个 Round
3. Round 内部消息按 timestamp 排序

**每个 Round 卡片**：

```
┌──────────────────────────────────────────────────────────┐
│  Round 2                                    ⚠️ D3 -10分  │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  ┌─ 推理 (reasoning) ─────────────────────────────────┐  │
│  │ 我需要再次获取项目代码确认最新状态...              │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌─ LLM 响应 ─────────────────────────────────────────┐  │
│  │ 让我获取项目代码来确认...                           │  │
│  │ [展开全文]                                          │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌─ 🔧 工具调用: git_pull ─── ⚠ 与上轮重复 ──────────┐  │
│  │ 输入: { "repo": "harbin", "branch": "main" }       │  │
│  │ ─────────                                           │  │
│  │ 输出: { "success": true, "commits": [...] }         │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌─ ❌ 扣分 ──────────────────────────────────────────┐  │
│  │ [D3] 连续两轮调用相同工具 · -10分                  │  │
│  │ 工具: git_pull                                      │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ⏱ 耗时 2.8s  ·  📊 tokens: in=980 out=156             │
└──────────────────────────────────────────────────────────┘
```

**卡片元素**：
- **卡片头部**：Round 编号 + 右侧扣分 badge（如果该轮有扣分）
- **推理过程**（如果有 `type=reasoning` 的 part）：斜体灰色背景块
- **LLM 响应**：`assistant` message 的 content，超过 5 行折叠
- **工具调用**：每个 `tool_call` part 展示 toolId + input JSON（可折叠）
- **工具结果**：对应 `tool_result` part 的 output JSON（可折叠）
- **扣分详情**：从 `deductions` 中筛选 `round` 匹配的记录，红色/橙色背景
- **底部统计**：耗时（从 step_start 到 step_finish 计算）、token 用量
- **中间注入的 system message**：如果某轮之间有新的 system message（中间纠正指令），用紫色边框标记

**JSON 折叠展示**：
- 默认折叠，显示一行预览（`JSON.stringify(obj).slice(0, 80) + '...'`）
- 展开后使用暗色代码块 `bg-slate-900 text-slate-100 font-mono text-[11px]`

### 7. Tab C：诊断分析

**核心价值**：让 LLM 根据完整的执行上下文自我分析错误原因。

**UI 结构**：

```
┌──────────────────────────────────────────────────────────┐
│  诊断分析                                                │
│                                                          │
│  ── 快速诊断 ──────────────────────────────────────────  │
│                                                          │
│  [为什么重复调用了相同工具?]  [为什么只拿到77分?]         │
│  [扣分记忆是否被有效利用?]   [哪些上下文影响了决策?]     │
│                                                          │
│  ── 自由提问 ──────────────────────────────────────────  │
│  ┌──────────────────────────────────────────┐  [发送]   │
│  │ 输入你想问的问题...                       │          │
│  └──────────────────────────────────────────┘           │
│                                                          │
│  ── 分析结果 ──────────────────────────────────────────  │
│  ┌────────────────────────────────────────────────────┐  │
│  │                                                    │  │
│  │ 根据对本次执行全过程的回溯分析：                   │  │
│  │                                                    │  │
│  │ 1. **Round 2 重复调用 git_pull 的根因**：          │  │
│  │    在 Round 1 的 git_pull 返回中，commits 列表     │  │
│  │    为空，LLM 判断需要重试...但 system prompt 中    │  │
│  │    的扣分记忆已明确警告过 D3 规则...               │  │
│  │                                                    │  │
│  │ 2. **扣分记忆的实际作用**：                        │  │
│  │    扣分记忆已注入（在 System Message #3 中），     │  │
│  │    但其位置在 Toolset Context 之后，被大量工具     │  │
│  │    定义稀释了注意力...                             │  │
│  │                                                    │  │
│  │ 3. **改进建议**：                                  │  │
│  │    - 将扣分记忆的注入位置前移...                   │  │
│  │    - 在任务指令中显式引用扣分规则...               │  │
│  │                                                    │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

**预设问题动态生成**：
- 根据 `scoreData.deductions` 存在的规则类型，动态生成相关问题
- 如有 D3 → 「为什么重复调用了相同工具?」
- 如有 D11 → 「为什么达到了最大轮次上限?」
- 如有 D8/D9 → 「为什么没有调用工具而是输出了纯文本?」
- 固定问题：「扣分记忆是否被有效利用?」「哪些上下文因素影响了 LLM 决策?」

**后端诊断 API**：`POST /agents/runtime/runs/:runId/diagnose`
- 请求体：`{ question: string }`
- 内部流程：
  1. 加载该 run 的 system messages（完整保留）
  2. 加载最近 N 轮交互（assistant + tool messages，截断避免 context 溢出）
  3. 加载评分 + 扣分详情
  4. 构造诊断 prompt：「你是一个 AI 系统的调试分析师。以下是一次 Agent 任务执行的完整记录...请回答用户的问题：{question}」
  5. 调用 LLM，流式返回（SSE）
- 响应：`text/event-stream`，前端逐 chunk 渲染

**分析结果展示**：
- 支持 Markdown 渲染（如果项目已有 markdown 渲染能力）
- 否则用 `whitespace-pre-wrap` 保持格式
- 加载中显示打字机效果的骨架屏

## 数据复用策略

**不需要新增任何前端数据加载逻辑**。现有 `useLogState` 中已经有：

| 数据 | 现有来源 | 决策回溯复用方式 |
|------|---------|-----------------|
| Run 基本信息 | `state.runItems[].run` | 直接传入抽屉 props |
| 评分数据 | `state.runScores[runId]` | 直接传入抽屉 props |
| 消息列表 | `state.runMessages[runId]` | 直接传入抽屉 props |

展开 Run 时 `toggleTaskExpanded` 已经同时触发 `loadRunScore` + `loadRunMessages`，数据在点击「决策回溯」前已经就绪。

唯一新增的数据加载：诊断分析 Tab 的 SSE 请求。

## 风险与注意事项

1. **Messages 数据量**：某些 Run 可能有几十轮、上百条 messages。执行时间线 Tab 应做虚拟滚动或分页，避免首次渲染卡顿
2. **诊断 API token 消耗**：需要截断策略。建议只保留全部 system messages + 最近 15 轮交互 + 评分摘要，控制在 8K~12K tokens 以内
3. **JSON 展示性能**：工具的 input/output 可能是大对象。折叠态只展示预览，展开时懒渲染
4. **z-index**：抽屉用 z-[48]，高于 Session 抽屉(z-40) 但不影响全局 Toast(z-120)
5. **Phase 1+2 不依赖后端**：可以先上线 Prompt 审查和执行时间线，诊断分析后续跟进
