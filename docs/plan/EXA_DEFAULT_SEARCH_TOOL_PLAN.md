# Exa 作为默认搜索工具实施计划

## 1. 背景与目标

- 需求：在当前项目中将 Exa 接入为默认搜索工具。
- 目标：在不破坏现有 Agent 定制能力的前提下，实现全局默认搜索工具切换为 Exa（`type=auto` + `highlights` 紧凑内容）。

## 2. 范围

- 后端搜索工具接入与默认路由策略。
- 配置与环境变量管理（`EXA_API_KEY`）。
- 相关文档同步（功能文档/配置说明/必要 API 说明）。
- 基础验证（类型检查或最小可运行校验）。

## 3. 执行步骤

1. 梳理现有搜索执行链路，确认默认搜索工具选择点与覆盖策略。
2. 增加 Exa 配置读取与校验逻辑，确保密钥仅从环境变量读取。
3. 实现 Exa 搜索 provider，并统一返回结构以兼容现有调用日志与上层消费。
4. 将 Exa 设为默认搜索工具，同时保留显式配置优先级（显式配置 > 全局默认）。
5. 更新相关文档（功能文档、配置说明、必要 API 文档），补充 `.env.example`（若缺失）。
6. 运行最小验证（lint/typecheck/关键路径自测）并记录结果。

## 4. 影响评估

- 后端：工具注册、默认选择策略、外部 API 调用逻辑。
- API：搜索工具默认行为变化（在未显式指定时切到 Exa）。
- 前端：工具管理页面默认展示可能受默认策略影响。
- 文档：需同步 Exa 默认策略与配置方式。

## 5. 风险与应对

- 未配置 `EXA_API_KEY`：默认搜索不可用。
  - 应对：增加启动期或运行期的明确报错与降级提示。
- 历史 Agent 已绑定特定搜索工具：可能与新默认冲突。
  - 应对：保持显式配置优先，不覆盖已有定制。
