# Scheduler Page Optimization Plan

## Goal

优化定时任务页面交互结构：进入页面默认展示定时计划列表，将“新建计划”收敛为弹窗流程，并通过右侧抽屉承载计划详情，提升信息密度与操作流畅度。

## Scope

- 定时任务管理页默认信息架构（前端）
- 新建定时计划弹窗化改造（前端）
- 定时计划详情右侧抽屉化展示（前端）
- 列表与详情联动、状态管理与交互反馈（前端）

## Plan

1. 盘点 `Scheduler` 页面现状与状态流，识别“列表/新建/详情”耦合点，明确需拆分的交互状态。
2. 调整页面主视图为“计划列表优先”，确保默认进入时可直接浏览定时计划并快速选择目标项。
3. 将“新建定时计划”表单迁移到弹窗，保留现有校验和提交逻辑，提交成功后自动刷新列表并聚焦新计划。
4. 将详情区域改为右侧抽屉，从列表项点击打开；在抽屉中展示计划概览、执行控制与历史记录。
5. 收口加载态、空态、错误态与并发交互（新建、启停、触发、删除）的一致性，避免状态串扰。
6. 完成前端自测，并检查相关功能文档是否需要更新。

## Impact

- Frontend: `frontend/src/pages/Scheduler.tsx`
- Frontend service: `frontend/src/services/schedulerService.ts`（按需，仅类型与调用复用）
- Backend/API: 无新增接口，复用现有 Scheduler API

## Risks / Dependencies

- 详情抽屉展示依赖 `getScheduleById` 返回字段完整度，需对缺省字段做前端兜底。
- 弹窗与抽屉并行时需处理状态隔离（如新建成功后详情刷新、关闭时机）。
- 删除当前选中计划后需保证选中状态与列表刷新一致，避免抽屉残留无效数据。
