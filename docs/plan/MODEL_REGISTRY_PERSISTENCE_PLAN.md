# 模型库持久化改造计划

## 需求理解

- 当前模型管理通过进程内数组保存新增模型，服务重启后数据会丢失。
- 目标是将模型管理改为 MongoDB 持久化，保证新增/删除/更新在重启后仍然生效。

## 执行步骤

1. 新增模型注册表 Schema，并在 `models` 模块中完成 Mongoose 注册。
2. 将 `ModelManagementService` 从内存数组改为数据库读写实现。
3. 增加服务启动种子逻辑：将 `AVAILABLE_MODELS` 幂等补齐到数据库。
4. 为模型唯一性增加约束，优先保证 `provider + model` 唯一，同时保留 `id` 唯一。
5. 调整模型管理控制器与工具调用链，兼容异步持久化读写。
6. 验证新增模型在重启后依然存在，并验证删除/更新路径。
7. 更新使用文档中的模型持久化行为说明。

## 关键影响点

- 后端：`apps/agents` 的 models 模块（service/controller/module/schema）。
- API：模型管理接口返回与时序（异步持久化）。
- 数据库：新增模型注册集合与唯一索引。
- 文档：用户指南中的模型管理行为说明。

## 风险与依赖

- 首次上线存在历史内存模型导入与重复冲突风险，需要幂等策略。
- 需要保证工具链调用结果与真实数据库状态一致，避免“口头成功”。
- lint 配置缺失时主要依赖构建与功能验证作为质量门禁。

## 验证方式

- 新增 `gpt-5.3-codex` 后重启服务，再次查询模型列表仍可见。
- 删除模型后重启服务，确认删除结果仍生效。
- `npm run build:agents` 通过。
