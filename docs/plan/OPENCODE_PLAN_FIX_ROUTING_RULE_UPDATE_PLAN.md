# OpenCode Plan/Fix 分流规则更新计划

## 1. 背景与目标

- 背景：当前协议默认所有开发需求先走 plan 确认与落盘流程，无法区分「新需求规划」与「问题修复」两类任务。
- 目标：建立基于用户输入的任务分流机制（plan/fix），保留 plan 原流程不变；fix 场景改为先询问是否落盘 fix 文档，并在开发完成后同步修复结果到 fix 文档。

## 2. 执行步骤

1. 盘点 `AGENTS.md` 中现有「需求处理流程协议」与「例外情况」条款，标记需要扩展的流程节点。
2. 新增任务类型判定规则：按用户输入判断任务为 `plan` 或 `fix`，并定义默认处理策略。
3. 保留 `plan` 场景原有执行顺序：先输出计划、等待同意、落盘 `docs/plan/`、再开始开发。
4. 新增 `fix` 场景执行顺序：先确认是否落盘 fix 文档（`docs/issue/fix/`），再执行修复；修复完成后同步写入修复结果。
5. 补充 fix 文档落盘要求（单 session 尽量单文档、文件名稳定可追踪、记录问题/根因/修复/验证结果）。
6. 校对与现有文档更新协议、dailylog 规则的一致性，避免冲突或重复约束。

## 3. 关键影响点

- OpenCode 需求处理流程（用户交互与执行时序）
- 文档落盘路径与命名规范（`docs/plan/`、`docs/issue/fix/`）
- 修复任务闭环要求（修复结果同步到 fix 文档）

## 4. 风险与依赖

- 用户描述可能同时包含规划与修复内容，需要优先按「当前主诉求」分流，必要时拆分为两段执行。
- 若用户明确要求跳过落盘，需尊重指令并在输出中说明跳过的文档动作。
- 需与既有「纯咨询可直接回答」例外规则保持一致，避免误触发文档流程。
