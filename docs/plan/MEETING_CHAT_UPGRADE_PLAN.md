# 会议/聊天升级优化计划

## 需求目标

围绕会议与聊天体验完成三项升级：支持打开独立新会议页面、聊天输入支持 `@成员` 智能提示、会议页面增加右侧操作区用于会议名称与参会人员管理。

## 执行步骤

1. 梳理现有会议/聊天页面结构、路由与成员数据流，确认新会议入口与消息输入组件的扩展点。
2. 实现“打开独立新会议页面”能力：补充路由与入口交互，支持从现有页面一键进入新会议会话。
3. 实现 `@成员` 提示：在消息输入框中识别触发词，展示成员候选列表，支持键盘选择与插入。
4. 实现右侧操作区：在会议页面增加侧栏，提供会议名称编辑与参会人员管理（查看、添加、移除）。
5. 完成前后端联调与状态同步，确保会议名称、成员列表与聊天 mention 数据一致。
6. 补充并更新相关测试与文档（README/docs/API 受影响项）。

## 关键影响点

- 前端：会议页面布局、路由、聊天输入组件、成员管理交互。
- 后端/API：会议名称修改与成员管理接口、成员查询能力。
- 数据模型：会议成员字段、成员展示信息与 mention 候选数据。
- 测试：会议创建跳转、`@成员` 提示、右侧管理区操作流程。
- 文档：功能说明与接口说明更新。

## 风险与依赖

- `@成员` 输入涉及光标定位与中文输入法兼容，需避免误触发和插入错位。
- 参会人员管理存在权限边界，需遵循当前主持人/管理员权限策略。
- 若现有接口未提供完整成员检索能力，需补充后端查询字段或新增接口。

## 需求修正记录

- 用户确认“单开页面”语义修正为：**某个会议可单独打开独立页面**，而非“会议模块整体新建页面”。
- 路由方案调整为：`/meetings/:meetingId` 直达指定会议；保留 `/meetings` 作为列表入口。
- 交互调整为：在会议列表项提供“单开”入口（新标签打开目标会议），移除“新会议页（/meetings/new）”语义。
- 页面形态调整为：`/meetings/:meetingId` 不展示全局菜单栏与会议列表，保留聊天区与可折叠会议操作区。
- 规则补充：当 1 对 1 会议新增非系统内置（非隐形）Agent 扩展为多人讨论时，会议名称需自动改为多人讨论语义。
- 交互补充：Agent 列表点击“开始聊天”后，应跳转会议页并自动打开目标会话（优先复用已有会话，不存在则创建）。
- 能力补充：未开始（pending）会议支持直接删除。

## Agent 思考状态后端化计划（2026-03-02）

### 需求目标

将 Agent“思考中”状态从前端推断改为后端权威状态：后端基于真实 responder 集合维护状态，Redis 存储短期态，前端通过 API 快照 + Socket 增量同步展示。

### 执行步骤

1. 在会议服务中新增 Agent 状态模型与事件类型，扩展 RedisService 读写能力（set/get/del/keys）。
2. Agent 被选为 responder 时写入 `thinking`；响应完成后写回 `idle`，并通过 `agent_state_changed` 事件推送。
3. 提供会议 Agent 状态快照接口（`GET /meetings/:id/agent-states`）。
4. 前端会议页接入状态快照轮询与 WebSocket 增量事件，移除本地“全员思考”猜测逻辑。
5. 完成前后端构建验证，确保编译通过。

- [x] 后端新增 Agent 状态事件类型 `agent_state_changed`，并在 responder 选中/响应完成时同步更新。
- [x] RedisService 新增 `set/get/del/keys` 能力，支持会议 Agent 状态短期存储与查询。
- [x] 新增快照接口 `GET /meetings/:id/agent-states`。
- [x] 前端会议页接入“快照轮询 + WS 增量”状态同步，移除本地全员思考推断。
