# 研发智能需求管理（Issue 协作）计划

## 1. 背景与目标

- 在 `研发智能` 模块内新增“需求管理”，承接 Agent 与人类讨论后的研发需求。
- 支持需求从 `todo` 状态开始，由上层 Agent（如 CTO）分发给研发 Agent。
- 研发 Agent 在执行过程中可持续更新状态，实时反映到“智能研发看板”。
- 打通 GitHub Issues，支持将内部需求一键同步为 GitHub Issue，并保留映射关系。

## 2. 执行步骤

1. 设计需求管理领域模型：需求主体、讨论记录、任务分发、看板事件、GitHub 映射。
2. 在 EI 后端新增需求管理接口：列表、详情、创建、状态流转、分发、讨论、看板查询。
3. 实现 GitHub Issues 一键同步能力：创建 Issue、回写链接、记录同步时间与错误。
4. 在主前端 `frontend/` 新增“需求管理”和“智能研发看板”页面，并接入相关 API。
5. 在需求详情中提供讨论流、分发动作、状态操作与 GitHub 同步入口。
6. 增加最小可用验证（类型检查/构建）并同步更新 feature/api/dailylog 文档。
7. 需求关联 EI 本地项目（local），并在同步 GitHub 时复用该项目绑定仓库；未绑定时明确禁用同步。

## 3. 关键影响点

- 后端（EI）：需求管理与看板聚合接口、GitHub 同步服务。
- 前端：需求列表、需求详情、看板页面与菜单路由。
- 数据库：新增需求集合与索引（状态、负责人、更新时间、GitHub 映射）。
- 数据库：需求新增 `localProjectId`，支持按本地项目筛选和索引。
- 文档：功能文档、API 文档、dailylog。

## 4. 风险与依赖

- GitHub API 调用依赖 `GITHUB_TOKEN` 权限与配额，需要错误提示与可重试。
- 内部状态与 GitHub `open/closed` 映射需保持单向主导，避免循环同步。
- 高并发更新下看板刷新可能出现短时延迟，首版采用轮询策略保证稳定。

## 5. 完成标准

- `研发智能 -> 需求管理` 页面可创建、筛选、分发、讨论和状态流转。
- `研发智能 -> 智能研发看板` 可按状态查看需求卡片并自动刷新。
- 需求可一键同步到 GitHub Issue，并在详情页展示映射链接。
- 需求列表支持“本地项目”筛选；需求创建可选所属本地项目；仅当项目已绑定 GitHub 时可执行同步。
- 需求详情支持删除动作（带二次确认），删除后刷新列表/看板缓存并返回需求列表页。
- 相关功能/API/日志文档按规范更新。
