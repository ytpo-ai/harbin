## 1. 需求理解与目标

当前系统工具形态存在两类并行：

- Composio 类（Slack/Gmail/GitHub）：以 `toolkit` 方式接入，单包包含多个方法。
- 系统内建类（memo/skill 等）：以“单工具-单能力”方式接入。

这会导致工具发现、权限治理、路由选择、监控审计和迭代发布口径不一致。  
本计划目标是在可控迁移窗口内完成全量重构，建立统一的工具元模型与执行治理链路，并下线旧体系。

### 目标

1. 对外统一调用单元：`Tool`（一个 action = 一个可执行工具）。
2. 对内保留管理边界：`Toolkit`（认证、配额、版本、Provider、生命周期）。
3. 建立统一 Registry、统一权限模型、统一执行与审计模型。
4. 在短暂迁移窗口内完成旧 id 映射与切流，窗口结束后移除旧体系。
5. 迁移完成后仅保留新模型，禁止新旧双轨长期并存。

### 非目标（本期不做）

- 不重写 Composio Provider 内部实现。
- 不保留长期兼容层与长期双轨运行。
- 不改变业务侧现有编排语义（meeting/orchestration 等）。

## 2. 当前问题清单

1. **发现口径不一致**：toolkit 方法与原子工具混用，LLM 侧选择成本高。
2. **权限粒度不一致**：有的按 toolkit 控制，有的按 tool 控制，审计粒度不统一。
3. **执行结果不统一**：分页、错误、超时、重试、异常码格式存在差异。
4. **治理能力分散**：限流、熔断、重试、幂等等策略缺少统一配置面。
5. **可观测性不足**：缺乏统一的“选中 -> 调用 -> 成败 -> 成本”链路指标。

## 3. 方案决策

采用 **统一模型全量改造方案（已确认）**：

- 管理层：`Toolkit`
- 执行层：`Tool`
- 连接层：`Adapter`

关键决策：

1. Composio 工具包在注册阶段展开为原子工具（例如 `github.issues.list`）。
2. 内建工具按同一命名规范纳入 Registry（例如 `internal.memo.search`）。
3. 所有调用方统一改造到新执行链，不再新增旧工具入口。
4. 执行管道统一加入：鉴权、参数校验、超时、重试、审计、指标。
5. 仅保留一个迭代周期的映射兼容层，之后强制下线。

## 7. 风险与依赖

### 风险

1. **命名冲突风险**：不同来源工具在统一命名下冲突。  
   缓解：引入 `provider.namespace.resource.action` 四段式命名。
2. **权限回归风险**：历史宽权限迁移到 action 级后行为变化。  
   缓解：先镜像审计，再切硬校验。
3. **流量抖动风险**：路由策略上线初期可能误选。  
   缓解：Top-K + fallback + 人工白名单兜底。

### 依赖

- 现有 `/tools` 调用方需支持最小兼容字段升级。
- 运行时日志链路需接入统一 `toolCallId/traceId`。
- 运维侧需提供指标看板与告警规则。

## 8. 回滚策略

1. 采用域级回滚开关（GitHub/Slack/Gmail/internal/mcp），支持分域回退。
2. 兼容窗口内允许回退到旧执行链；窗口结束后仅允许版本回滚，不恢复双轨。
3. 发生权限异常时优先回退策略层，再回退执行器版本。

## 9. alias 下线判定规则

1. 使用 `GET /tools/registry/alias-hits` 作为唯一观测口径。
2. 判定阈值调整为：连续 **1 天** alias 命中为 0。
3. 达到阈值后执行：关闭 alias 映射开关 -> 发布下线通知 -> 保留 1 天只读观察。
4. 若观察期出现兼容请求，仅允许版本回滚，不恢复长期双轨。
