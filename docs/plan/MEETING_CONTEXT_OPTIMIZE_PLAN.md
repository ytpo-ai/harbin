# 会议上下文去噪与 Prompt Registry 能力建设计划

## 背景与目标

当前会议场景存在系统消息冗余注入、执行闭环不稳定、Prompt 仅依赖代码硬编码的问题。目标是通过分步优化，在不改变核心业务流程的前提下，提升会话质量、执行效率与 Prompt 可运营性。

## 已识别关键问题

1. 用户身份 identify 相关 system 内容在同一会话被反复注入，造成上下文噪音与 token 浪费。
2. 固定格式任务信息（如标题/描述/优先级块）在无变化时重复注入，信息增量价值低。
3. 指令重复确认偏多（确认人选 -> 确认文案 -> 确认发送），降低会议执行效率。
4. 存在空响应风险（异常时输出 `-`），缺乏最小可用兜底回复。
5. “已执行”缺少可追踪证据（未同步 requirement 状态/通知摘要/检查点）。
6. 会议节奏被微小确认打断，未形成“1 轮闭环、最晚 2 轮”执行标准。

## 流程优化原则（orchestration-planner-guard 轻量版）

- Plan：会议分配类任务固定拆为两个原子动作（分配 + 通知）。
- Guard：仅在关键歧义时提问；无歧义场景直接执行。
- Execute：先落 requirement 状态，再发送通知，避免“口头已发、系统未落地”。
- Report：统一使用结构化回执，一次性同步结果、证据与下一检查点。

## 执行步骤

### Step 1：会议执行链路优化（系统消息去重 + 执行闭环）

1. 对 identity 与 task-info system block 引入 fingerprint + delta 注入策略：内容无变化则不重复写入。
2. 建立“一次确认后自动执行”规则：用户明确同意后直接执行，不再二次询问语气/文案。
3. 强制三段式回执模板：
   - 动作 1：requirement 已 assign（需求 ID、执行人、状态）
   - 动作 2：已通知（消息摘要）
   - 下一步：预计检查时间点（如今日 EOD）
4. 增加空响应兜底：生成异常时自动返回“操作进行中，1 分钟内补充回执”并触发重试。
5. 通知文案模板化：默认短版开工通知，除非明确要求再附加验收细节。
6. 会议 SLA：分配类动作目标 1 轮闭环，最晚不超过 2 轮。

### Step 2：Prompt Registry 后端能力（DB/Redis 优先）

1. 新增 PromptResolver，解析优先级：session override > DB(published) > Redis cache > code default。
2. 会议场景与计划编排场景接入 Resolver，替代单一硬编码来源。
3. 设计模板实体字段：scene、role、version、status、content、updatedBy、updatedAt。
4. 缓存策略：发布后失效 Redis 并回填，保证读取一致性与回滚可用性。

### Step 3：Prompt 管理能力（前端页面 + 发布链路）

1. 提供管理页面：查看、编辑、发布、回滚、版本对比（diff）、生效范围管理（scene/role/env）。
2. 提供发布与回滚接口，并记录完整审计日志（操作者、时间、版本、变更摘要）。
3. 支持会议场景/计划编排场景的固定 Prompt 可视化维护，满足长期优化诉求。

### Step 4：安全与稳定保护（发布前校验 + 快速回退）

1. 发布前校验：变量占位符合法性、内容长度、基础敏感词与格式规则。
2. 引入 ReleaseGuard：不满足校验条件时拦截发布。
3. 建立一键回滚机制：异常时快速恢复上一稳定版本并刷新缓存。
4. 增加关键监控：重复注入率、空响应次数、回执完整率、分配闭环轮次。

## 关键影响点

- 后端：会议消息编排、PromptResolver、发布/回滚/审计 API。
- 前端：Prompt 管理页面与发布交互。
- 数据层：Prompt 模板持久化（DB）与热缓存（Redis）。
- 运维与质量：监控告警、回滚机制、执行闭环验证。

## 风险与依赖

1. 风险：去重策略过严可能误删必要系统消息。
   - 缓解：仅对同会话同类型且 fingerprint 一致内容去重，保留增量变更注入。
2. 风险：错误 Prompt 发布导致场景级质量下降。
   - 缓解：发布前校验 + 发布拦截 + 一键回滚 + 审计追踪。
3. 风险：缓存不一致造成新旧 Prompt 混用。
   - 缓解：发布后统一失效与回填，读取侧增加版本校验。

## 完成判定标准

1. 会议场景中 identity/task-info 无变化时不再重复注入。
2. 分配类任务回执均包含“已分配 + 已通知 + 下一检查点”。
3. 不再出现空响应 `-`，异常场景有兜底回复。
4. 系统 Prompt 可通过 DB/Redis 与前端页面管理，且支持发布与回滚。
